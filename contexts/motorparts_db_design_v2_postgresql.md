# Motorparts POS Database Documentation (PostgreSQL-Compatible)

## 1. Purpose

This document converts the existing MySQL-oriented design in `motorparts_db_design_v2` into a PostgreSQL-compatible implementation guide while preserving all business rules:

- Single-store inventory model
- Parts-only catalog in-system (services are handled outside this system)
- Trigger-driven stock movement ledger
- Supplier cost history automation
- RBAC with action-level permissions
- Receipt/OR-SI sequence management

Target PostgreSQL version: **PostgreSQL 14+** (works best on 15/16).

---

## 2. PostgreSQL Compatibility Strategy

### 2.1 Type and Syntax Mapping

| MySQL/MariaDB | PostgreSQL Equivalent | Notes |
|---|---|---|
| `BIGINT AUTO_INCREMENT` | `BIGINT GENERATED BY DEFAULT AS IDENTITY` | Preferred modern PK style |
| `INT AUTO_INCREMENT` | `INTEGER GENERATED BY DEFAULT AS IDENTITY` | |
| `TINYINT(1)` | `BOOLEAN` | Use `TRUE/FALSE` instead of `1/0` |
| `ENUM(...)` | `CREATE TYPE ... AS ENUM (...)` | Can also use `TEXT + CHECK`, but enum type is cleaner |
| `JSON` | `JSONB` | Better indexing/performance for queryable JSON |
| `TIMESTAMP` | `TIMESTAMPTZ` or `TIMESTAMP` | Use `TIMESTAMPTZ` for audit/sales events |
| `ON UPDATE CURRENT_TIMESTAMP` | Trigger-based `updated_at` maintenance | PostgreSQL has no direct column clause |
| `CURDATE()` | `CURRENT_DATE` | |
| `CURRENT_TIMESTAMP` | `CURRENT_TIMESTAMP` / `NOW()` | Both valid |
| `DELIMITER $$ ...` | Not needed | PostgreSQL uses dollar-quoted functions (`$$`) directly |
| `SIGNAL SQLSTATE '45000'` | `RAISE EXCEPTION` | For business rule failures |
| `CONCAT(a,b,c)` | `a || b || c` | String concatenation |

### 2.2 Behavioral Differences to Handle

1. `updated_at` auto-maintenance requires a trigger per table.
2. Session variable `@current_user_id` should be replaced with PostgreSQL session config:
   - `SELECT set_config('app.current_user_id', '123', false);`
   - Read in trigger via `current_setting('app.current_user_id', true)`.
3. Generated columns are supported, but must use immutable/stable-safe expressions.
4. Keep decimal precision exactly for financial fields (`NUMERIC(14,4)`, etc.).

### 2.3 Scope Rule

Services (labor jobs) are out of scope for this schema. POS transactions and inventory logic in this design are for physical parts/accessories only.

## 3.2 Generic `updated_at` Trigger

```sql
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
  NEW.updated_at := CURRENT_TIMESTAMP;
  RETURN NEW;
END;
$$;
```

Apply this trigger to every table that has `updated_at`.

Example:

```sql
CREATE TRIGGER trg_products_set_updated_at
BEFORE UPDATE ON products
FOR EACH ROW
EXECUTE FUNCTION set_updated_at();
```

## 4.2 `product_variants`

```sql
CREATE TABLE product_variants (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    product_id      BIGINT NOT NULL,
    variant_sku     VARCHAR(100) NOT NULL UNIQUE,
    variant_name    VARCHAR(200) NULL,
    attributes      JSONB NULL,
    cost_price      NUMERIC(12,4) NULL,
    selling_price   NUMERIC(12,4) NULL,
    wholesale_price NUMERIC(12,4) NULL,
    margin_pct      NUMERIC(8,4) GENERATED ALWAYS AS (
      CASE
        WHEN COALESCE(selling_price, 0) > 0
          THEN ROUND(((selling_price - COALESCE(cost_price, 0)) / selling_price) * 100, 4)
        ELSE 0
      END
    ) STORED,
    markup_pct      NUMERIC(8,4) GENERATED ALWAYS AS (
      CASE
        WHEN COALESCE(cost_price, 0) > 0
          THEN ROUND(((COALESCE(selling_price, 0) - cost_price) / cost_price) * 100, 4)
        ELSE 0
      END
    ) STORED,
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at      TIMESTAMPTZ NULL,
    CONSTRAINT fk_product_variants_product
      FOREIGN KEY (product_id) REFERENCES products(id)
      ON DELETE CASCADE ON UPDATE CASCADE
);

CREATE INDEX idx_product_variants_product ON product_variants(product_id);
CREATE INDEX idx_pv_margin_pct ON product_variants(margin_pct);
```

## 5. Trigger and Procedure Conversions (MySQL -> PostgreSQL)

## 5.1 Stock Deduction Trigger (`sales_transaction_items`)

```sql
CREATE OR REPLACE FUNCTION trg_sti_after_insert_fn()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_qty_before NUMERIC(12,4);
  v_qty_after  NUMERIC(12,4);
  v_cashier_id BIGINT;
BEGIN
  SELECT cashier_id
    INTO v_cashier_id
  FROM sales_transactions
  WHERE id = NEW.sales_transaction_id;

  SELECT qty_on_hand
    INTO v_qty_before
  FROM inventory_stock
  WHERE product_variant_id = NEW.product_variant_id
  FOR UPDATE;

  IF v_qty_before < NEW.qty THEN
    RAISE EXCEPTION 'Insufficient stock for this product';
  END IF;

  v_qty_after := v_qty_before - NEW.qty;

  UPDATE inventory_stock
     SET qty_on_hand = v_qty_after,
         updated_at = CURRENT_TIMESTAMP
   WHERE product_variant_id = NEW.product_variant_id;

  INSERT INTO stock_movements (
    product_variant_id, movement_type, reference_type,
    reference_id, qty_before, qty_change, qty_after,
    unit_cost, performed_by, created_at
  ) VALUES (
    NEW.product_variant_id,
    'sale', 'sales_transaction',
    NEW.sales_transaction_id,
    v_qty_before, -NEW.qty, v_qty_after,
    NEW.unit_cost, v_cashier_id, CURRENT_TIMESTAMP
  );

  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_sti_after_insert
AFTER INSERT ON sales_transaction_items
FOR EACH ROW
EXECUTE FUNCTION trg_sti_after_insert_fn();
```

## 5.2 Stock Restoration Trigger (`sales_return_items`)

```sql
CREATE OR REPLACE FUNCTION trg_sri_after_insert_fn()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_qty_before NUMERIC(12,4);
  v_qty_after  NUMERIC(12,4);
  v_cashier_id BIGINT;
BEGIN
  IF NEW.restock = TRUE THEN
    SELECT cashier_id
      INTO v_cashier_id
    FROM sales_returns
    WHERE id = NEW.sales_return_id;

    SELECT qty_on_hand
      INTO v_qty_before
    FROM inventory_stock
    WHERE product_variant_id = NEW.product_variant_id
    FOR UPDATE;

    v_qty_after := v_qty_before + NEW.qty_returned;

    UPDATE inventory_stock
       SET qty_on_hand = v_qty_after,
           updated_at = CURRENT_TIMESTAMP
     WHERE product_variant_id = NEW.product_variant_id;

    INSERT INTO stock_movements (
      product_variant_id, movement_type, reference_type,
      reference_id, qty_before, qty_change, qty_after,
      unit_cost, performed_by
    ) VALUES (
      NEW.product_variant_id,
      'sale_return', 'sales_return',
      NEW.sales_return_id,
      v_qty_before, NEW.qty_returned, v_qty_after,
      NEW.unit_price, v_cashier_id
    );
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_sri_after_insert
AFTER INSERT ON sales_return_items
FOR EACH ROW
EXECUTE FUNCTION trg_sri_after_insert_fn();
```

## 5.3 PO Receipt Trigger (`purchase_order_items`)

```sql
CREATE OR REPLACE FUNCTION trg_poi_after_update_fn()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_qty_before   NUMERIC(12,4);
  v_qty_received NUMERIC(12,4);
  v_qty_after    NUMERIC(12,4);
  v_old_avg_cost NUMERIC(14,6);
  v_new_avg_cost NUMERIC(14,6);
  v_created_by   BIGINT;
BEGIN
  v_qty_received := NEW.qty_received - OLD.qty_received;

  IF v_qty_received > 0 THEN
    SELECT created_by
      INTO v_created_by
    FROM purchase_orders
    WHERE id = NEW.purchase_order_id;

    SELECT qty_on_hand, avg_cost
      INTO v_qty_before, v_old_avg_cost
    FROM inventory_stock
    WHERE product_variant_id = NEW.product_variant_id
    FOR UPDATE;

    v_new_avg_cost := ((v_qty_before * v_old_avg_cost) + (v_qty_received * NEW.unit_cost))
                      / NULLIF(v_qty_before + v_qty_received, 0);

    v_qty_after := v_qty_before + v_qty_received;

    UPDATE inventory_stock
       SET qty_on_hand = v_qty_after,
           avg_cost = COALESCE(v_new_avg_cost, avg_cost),
           updated_at = CURRENT_TIMESTAMP
     WHERE product_variant_id = NEW.product_variant_id;

    INSERT INTO stock_movements (
      product_variant_id, movement_type, reference_type,
      reference_id, qty_before, qty_change, qty_after,
      unit_cost, performed_by
    ) VALUES (
      NEW.product_variant_id,
      'purchase_receipt', 'purchase_order',
      NEW.purchase_order_id,
      v_qty_before, v_qty_received, v_qty_after,
      NEW.unit_cost, v_created_by
    );

    IF NEW.unit_cost <> OLD.unit_cost OR OLD.qty_received = 0 THEN
      INSERT INTO supplier_cost_history (
        supplier_id, product_variant_id, supplier_product_id,
        old_unit_cost, new_unit_cost, change_source,
        purchase_order_id, change_reason, changed_by, effective_date
      )
      SELECT
        po.supplier_id,
        NEW.product_variant_id,
        sp.id,
        COALESCE(OLD.unit_cost, NEW.unit_cost),
        NEW.unit_cost,
        'po_receipt',
        NEW.purchase_order_id,
        'Auto: PO receipt - ' || po.po_number,
        v_created_by,
        CURRENT_DATE
      FROM purchase_orders po
      LEFT JOIN supplier_products sp
        ON sp.supplier_id = po.supplier_id
       AND sp.product_variant_id = NEW.product_variant_id
      WHERE po.id = NEW.purchase_order_id;
    END IF;
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_poi_after_update
AFTER UPDATE ON purchase_order_items
FOR EACH ROW
EXECUTE FUNCTION trg_poi_after_update_fn();
```

## 5.4 Manual Supplier Cost Change Trigger (`supplier_products`)

```sql
CREATE OR REPLACE FUNCTION trg_sp_cost_history_fn()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_changed_by BIGINT;
BEGIN
  IF NEW.unit_cost <> OLD.unit_cost THEN
    v_changed_by := COALESCE(NULLIF(current_setting('app.current_user_id', true), '')::BIGINT, 0);

    INSERT INTO supplier_cost_history (
      supplier_id, product_variant_id, supplier_product_id,
      old_unit_cost, new_unit_cost, change_source,
      change_reason, changed_by, effective_date
    ) VALUES (
      NEW.supplier_id, NEW.product_variant_id, NEW.id,
      OLD.unit_cost, NEW.unit_cost,
      'manual_update', NULL, v_changed_by, CURRENT_DATE
    );
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_sp_cost_history
AFTER UPDATE ON supplier_products
FOR EACH ROW
EXECUTE FUNCTION trg_sp_cost_history_fn();
```

## 5.5 Stored Procedure Conversion (`sp_complete_sale`)

In PostgreSQL, this can be a function returning a result row:

```sql
CREATE OR REPLACE FUNCTION sp_complete_sale(
  p_transaction_id BIGINT,
  p_amount_tendered NUMERIC(14,4),
  p_payment_method_id SMALLINT
)
RETURNS TABLE (
  change_given NUMERIC(14,4),
  result_code SMALLINT,
  result_message VARCHAR(200)
)
LANGUAGE plpgsql
AS $$
DECLARE
  v_total_amount NUMERIC(14,4);
  v_session_status cash_session_status_enum;
BEGIN
  SELECT cs.status
    INTO v_session_status
  FROM sales_transactions st
  JOIN cash_sessions cs ON cs.id = st.cash_session_id
  WHERE st.id = p_transaction_id
  FOR UPDATE;

  IF v_session_status <> 'open' THEN
    RAISE EXCEPTION 'Cash session is not open';
  END IF;

  SELECT total_amount
    INTO v_total_amount
  FROM sales_transactions
  WHERE id = p_transaction_id
    AND status = 'pending'
  FOR UPDATE;

  IF v_total_amount IS NULL THEN
    RAISE EXCEPTION 'Transaction not found or already processed';
  END IF;

  IF p_amount_tendered < v_total_amount THEN
    RAISE EXCEPTION 'Insufficient payment amount';
  END IF;

  change_given := p_amount_tendered - v_total_amount;

  UPDATE sales_transactions
     SET status = 'completed',
         amount_tendered = p_amount_tendered,
         change_given = change_given
   WHERE id = p_transaction_id;

  INSERT INTO transaction_payments (sales_transaction_id, payment_method_id, amount)
  VALUES (p_transaction_id, p_payment_method_id, p_amount_tendered);

  result_code := 0;
  result_message := 'Sale completed successfully';

  RETURN NEXT;
EXCEPTION
  WHEN OTHERS THEN
    change_given := 0;
    result_code := 1;
    result_message := 'Transaction failed: ' || SQLERRM;
    RETURN NEXT;
END;
$$;
```

## 6. PostgreSQL Index and Performance Notes

1. Keep all existing composite indexes from the MySQL design.
2. Add partial indexes for active rows when useful:

```sql
CREATE INDEX idx_products_active_catalog
ON products (category_id, sku)
WHERE is_active = TRUE AND deleted_at IS NULL;
```

3. For very large `stock_movements` and `sales_transactions`, consider monthly range partitioning on `created_at` / `transaction_date`.
4. Use `JSONB` GIN indexes only for attributes queried frequently.

## 7. Migration Checklist (MySQL -> PostgreSQL)

1. Create enum types first.
2. Create tables with identity columns and PostgreSQL data types.
3. Add foreign keys, check constraints, and indexes.
4. Add `set_updated_at` trigger to all tables with `updated_at`.
5. Create business triggers (`trg_sti_after_insert`, `trg_sri_after_insert`, `trg_poi_after_update`, `trg_sp_cost_history`).
6. Create reporting views (`v_current_stock`, `v_low_stock_alerts`, `v_daily_sales_summary`, `v_sales_performance`, `v_supplier_cost_trends`) using PostgreSQL syntax.
7. Seed RBAC data (`roles`, `permissions`, `role_permissions`) and baseline users.
8. In the application session, set current user id before writes that require auditing:

```sql
SELECT set_config('app.current_user_id', '1', false);
```

9. Validate end-to-end scenarios:
   - Part sale deducts stock and writes stock movement.
  - Return restocks when `restock = true`.
   - PO receipt updates stock + weighted avg cost + supplier cost history.
